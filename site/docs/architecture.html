Deep scan workflow

  1. Policy Load. Parse and validate sentry.yaml. Resolve target namespaces.
  2. Catalog Resolution. Connect (HMS Thrift, Glue API, REST, file walk) and fetch each table's metadata location.
  3. Metadata Snapshot Lock. Download vN.metadata.json. Lock subsequent operations to the locked snapshot ID — in-flight commits are invisible.
  4. Write-pattern classification. Analyse commit-interval CV across the trailing 50 snapshots.
  5. Manifest walk:
    • Worker A parses the manifest list → dispatches manifest file URIs.
    • Workers B–Z parse manifests concurrently (Avro, streaming) → collect (size, partition, content) per file.
  6. Storage crawl (orphan mode only). Streaming list with bloom-filter pre-check.
  7. Diagnostic analysis. Feed stats into the health engine; compute per-dimension scores.
  8. PII sampling (when invoked). Worker pool reads Parquet footers, samples row groups, runs in-memory regex + entropy. Zero disk writes.
  9. Output generation. Format per --format; apply policy thresholds for the exit code.

Selected design decisions

Scale without OOM (10M+ file tables)

The orphan-tracker uses a double-hashed Bloom filter (Kirsch–Mitzenmacher) sized for the expected file count at a 0.1% FPR. For genuinely huge tables, the interface accepts a disk-backed implementation (bbolt/pebble) — that's a Phase 4 add. Streaming Avro parsing means we never hold a full manifest in memory.

Point-in-time consistency

Every command captures the snapshot ID at the start; all subsequent storage and metadata reads pin to that snapshot. This is exactly the guarantee Iceberg's snapshot isolation already provides — we just respect it across the entire scan.

PII without persistence

Go's io.Reader plumbing and the Parquet library's page.Values() iterator let us walk values without materialising columns. Each value is regex-matched and immediately discarded. Column names and detected PII types are the only data that reaches output. strace-verifiable.

Accurate delete amplification

For each data file we count associated position-delete and equality-delete files from the manifest entries. The amplification estimate follows the Iceberg spec's merge-on-read cost model. Conservative — actual amplification depends on engine implementation.

Write-pattern accuracy

The classifier analyses the last 50 snapshots using commit-interval variance and files-per-commit distribution. ~90% accuracy across the test corpus; the write_pattern override in sentry.yaml handles the edge cases.

Multi-cloud cost modelling

Cost rates live behind a cost.Provider interface. Default provider uses S3 Standard list-price rates. Custom providers loadable via YAML config in a future release.
