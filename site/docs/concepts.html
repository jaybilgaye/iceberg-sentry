Concepts

Everything you need to know to read the output of every Sentry command.

Iceberg table anatomy

An Iceberg table is three layers:

  1. Catalog — a key-value store mapping namespace.table to a metadata pointer. Glue, Hive Metastore, Iceberg REST, Polaris, Unity, or a directory on disk all qualify.
  2. Metadata — a JSON file (vN.metadata.json) holding the table's schema history, partition specs, snapshot log, and refs.
  3. Manifests — Avro files. The manifest list for a snapshot points at one or more manifest files, each of which describes a set of data files (and, in v2, delete files).

Sentry walks all three. The catalog gives us the metadata location; the metadata JSON gives us snapshots and the manifest-list URI; each manifest file enumerates data + delete files with their sizes, partition values, and content type. That's enough to score health without touching a Parquet byte.

The health score

Every audit emits a composite 0–100 score (currently summed to 90 because schema_evolution is Phase 4). Dimensions and their max weights:

DimensionMaxWhat it measures
file_size20Small-file ratio (under 64 / 128 MB) vs. data-file count.
delete_amplification20Position + equality delete files vs. data files. v2 only.
manifest_density15Manifest count vs. policy threshold (default 2k warn, 5k crit).
snapshot15Snapshot count + age of oldest snapshot.
partition_skew15Coefficient of variation across per-partition bytes; hot/sparse counts.
format_version5v1 surfaces as INFO; v2 + v3 are full marks.

Thresholds are tunable per-namespace via sentry.yaml.

Severity model

OKDimension within policy. No action required.
INFOInformational signal (e.g. v1 format).
WARNINGApproaching threshold. Surface in CI with --fail-on warn.
CRITICALThreshold breached. Default --fail-on critical fails the build.

The worst severity across dimensions determines the table's overall severity and the process exit code.

Write-pattern classification

A streaming Flink job that emits files every minute should not be flagged as "small file syndrome" — that's its normal state. Sentry classifies tables based on the trailing 50 snapshots:

  • streaming — median inter-commit interval ≤ 5 minutes
  • batch — median inter-commit interval ≥ 1 hour
  • mixed — CV of intervals > 1.0 (clear bimodal distribution)
  • unknown — fewer than two snapshots

The pattern is reported in every output and affects scoring: streaming tables get the file-size warn threshold relaxed by 4×.

Snapshot lock — point-in-time consistency

Concurrent writers can append snapshots while Sentry is scanning. Every command locks to a specific snapshot ID at start and operates against that snapshot for the entire run. In-flight commits after the lock are invisible to the audit, so partial-write states never affect results.

How each dimension is computed

file_size

  • Fraction of data files under min_file_size_mb (default 128 MB).
  • ≥ 50% small → CRITICAL (score 8). 25–50% → WARNING (score 12). Else OK.
  • Streaming pattern divides the threshold by 4 (floor 16 MB).

delete_amplification

  • v2 tables only. Counts position + equality delete files separately.
  • ratio = (position + equality) / data file count.
  • ≥ delete_file_ratio_fail (default 0.25) → CRITICAL.
  • ≥ delete_file_ratio_warn (default 0.10) → WARNING.

manifest_density

  • Total manifest file count.
  • ≥ crit (default 5,000) → CRITICAL.
  • ≥ warn (default 2,000) → WARNING.

snapshot

  • Snapshot count + age of the oldest snapshot.
  • oldest ≥ 90d or count ≥ 500 → CRITICAL.
  • oldest ≥ 30d or count ≥ 100 → WARNING.

partition_skew

  • Coefficient of variation across per-partition byte volume.
  • ≥ 1.0 → CRITICAL. ≥ 0.5 → WARNING.
  • Also reports "hot" (> 2× median) and "sparse" (< 10% median) counts.

format_version

  • v1 → INFO (recommend upgrade to v2 for row-level deletes).
  • v2, v3 → OK.

Catalog vs. storage

Catalogs and storage backends are independent:

  • Catalog answers "where is this table's metadata?" via a JDBC/Thrift/HTTP call.
  • Storage answers "give me the bytes of this URI." S3, ADLS Gen2 (via S3-compatible API), HDFS (WebHDFS), local FS.

The same Glue catalog can point at S3 paths in three different buckets and three different regions; the storage layer dispatches by URI scheme.

Zero-persistence PII scanning

The PII scanner is built so column values never escape the byte stream. The Parquet row-group reader pipes values directly into the regex/entropy classifier and discards them immediately. Only column names and detected PII types are written to the report. Confirm with strace if you need to.
