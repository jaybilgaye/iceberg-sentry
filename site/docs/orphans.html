orphans

Find files in storage that no valid Iceberg snapshot references. Dry-run by default. Safety controls built in.

// safety guarantee Iceberg Sentry never deletes files. The --confirm flag is reserved for a future destructive mode; the current implementation always emits a recommendation manifest.

How it works

  1. Lock the scan to a specific Iceberg snapshot.
  2. Walk every snapshot in metadata. For each snapshot, fetch the manifest list and every manifest file. Insert every referenced file URI into an exact set and a Bloom filter.
  3. Stream object listings from storage under the table's data prefix.
  4. Any object that isn't in the exact set and whose mtime is older than the grace period is an orphan candidate.

Examples

Dry-run with default 24h grace

$ iceberg-sentry orphans \
    --catalog rest --rest http://catalog:8181 \
    --table finance.transactions

Aggressive 6h grace, JSON output for a deletion pipeline

$ iceberg-sentry orphans \
    --catalog glue --table finance.transactions \
    --grace-period 6h --format json > orphans.json

Crawl a non-default data prefix

$ iceberg-sentry orphans \
    --catalog rest --rest http://catalog:8181 \
    --table finance.transactions \
    --data-prefix s3://datalake/finance/transactions/data

Flags

FlagTypeDefaultDescription
--tablestringrequiredFully qualified table id.
--data-prefixstring<table>/dataStorage prefix to crawl.
--grace-periodduration24hSkip files newer than this. 6h, 30m, etc.
--formatstringtexttext · json
--previewint25Max candidates to list inline (0 = all).
--confirmboolfalseReserved for future destructive mode (no-op today).

Catalog and storage flags (--catalog, --rest, --hive, --hdfs, --s3-path-style) are identical to audit.

Memory footprint

The active-file exact set is held in memory; a 1M-file table uses approximately 250 MB for the set. Disk-backed swap (bbolt) is on the Phase 4 roadmap for tables > 10M files.
