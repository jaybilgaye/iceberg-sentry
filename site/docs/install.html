Installation

Iceberg Sentry is distributed as a single static binary (< 20MB) and a Docker image. Build from source via Go is also supported.

$ curl -sSL https://raw.githubusercontent.com/jaybilgaye/iceberg-sentry/main/scripts/install.sh | bash
$ iceberg-sentry version
iceberg-sentry 0.3.0 phase-3

Installs into $HOME/.local/bin. Override the target with SENTRY_PREFIX=/usr/local bash.

$ brew tap jaybilgaye/iceberg-sentry
$ brew install iceberg-sentry
$ docker run --rm \
    -v $PWD/warehouse:/data:ro \
    ghcr.io/jaybilgaye/iceberg-sentry:latest \
    audit --catalog localfs --catalog-root /data --table finance.transactions

Mount catalog credentials (AWS profile, keytab, etc.) as read-only volumes. The image runs as UID 65534 by default.

$ helm install sentry deploy/helm \
    --set cronjob.enabled=true \
    --set exporter.enabled=true \
    --set serviceMonitor.enabled=true

See Deployment for the full chart reference.

$ git clone https://github.com/jaybilgaye/iceberg-sentry.git
$ cd iceberg-sentry
$ go build -o iceberg-sentry ./cmd/iceberg-sentry
$ ./iceberg-sentry version

Requires Go 1.23+. The repo contains no submodules; go build handles everything.

Verify the install

$ iceberg-sentry --help
iceberg-sentry audits Apache Iceberg tables for physical, metadata,
performance, and cost-efficiency issues without requiring a running
compute engine.

Usage:
  iceberg-sentry [command]

Available Commands:
  audit       Audit one or more Iceberg tables and report a health score
  bench       Capture a baseline scan and compare against a fresh scan
  cost        Snapshot cost timeline and tiered-storage recommendations
  export      Long-lived Prometheus exporter (serves /metrics)
  migration   On-prem HDFS → CDP Public Cloud Migration Readiness Audit
  orphans     Report files in storage that no valid Iceberg snapshot references
  pii         Sample Parquet row groups for PII and emit catalog-import findings
  version     Print the iceberg-sentry version

System requirements

  • Linux (amd64, arm64), macOS (arm64, amd64), Windows (amd64)
  • No JVM required.
  • Memory: ≤ 256 MB for typical tables, ≤ 512 MB for 10M+ file tables (disk-backed Bloom kicks in > 500k active files — Phase 4).
  • Network access only to your catalog + storage endpoints.
